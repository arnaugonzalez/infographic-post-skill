# Requirements: Infographic Skill

**Defined:** 2026-03-23
**Milestone:** v1.0 — Multi-Provider Model Support
**Core Value:** Turn any data or context into a publication-ready infographic with one command — no design tools required.

## v1.0 Requirements

### Provider Configuration (PROV)

- [x] **PROV-01**: User can set `INFG_LLM_PROVIDER=openrouter` in `.env` to route all text generation through OpenRouter instead of Gemini
- [x] **PROV-02**: User can set `INFG_LLM_MODEL` to any model string (e.g. `anthropic/claude-opus-4`) to override the default text model for HTML-output path
- [x] **PROV-03**: User can set `INFG_LLM_API_KEY` as a generic LLM provider API key, used when no provider-specific key is configured
- [x] **PROV-04**: User can set `INFG_OPENROUTER_API_KEY` as a dedicated OpenRouter API key that takes precedence over `INFG_LLM_API_KEY` when provider is openrouter
- [x] **PROV-05**: User can set `INFG_IMAGE_MODEL` to override the default image model (`gemini-3.1-flash-image-preview`) without changing code

### OpenRouter Integration (OROUTER)

- [x] **OROUTER-01**: User can generate infographics via the HTML-output path using any LLM available on OpenRouter when `INFG_LLM_PROVIDER=openrouter`
- [x] **OROUTER-02**: User sees a clear, actionable error message when OpenRouter API key is invalid (401) or account has insufficient credits (402)
- [x] **OROUTER-03**: User sees a validation error with correct format example if model name is missing the required `provider/model` format before any API call is made
- [x] **OROUTER-04**: User sees token usage counts (input/output tokens) after a successful OpenRouter run in the cost report section
- [x] **OROUTER-05**: User can override provider and model per-invocation via `--llm-provider` and `--llm-model` CLI flags (higher precedence than env vars)

### Deploy Readiness (DEPLOY)

- [x] **DEPLOY-01**: OpenRouter API keys (`sk-or-v1-...` format) are redacted in all error output, log messages, and tracebacks — matching the existing Google key redaction behavior
- [x] **DEPLOY-02**: Every env var consumed by `os.environ.get()` in all scripts is documented in `.env.example` with a description and example value
- [x] **DEPLOY-03**: SKILL.md and README.md document the OpenRouter setup path and correct the Python version requirement from 3.8 to 3.9+
- [x] **DEPLOY-04**: Infographic generation via the matplotlib offline path works correctly when the `openai` package is not installed (lazy import guard)

## v2 Requirements

### Extended Provider Support

- **EXTPROV-01**: User can configure a custom image generation provider beyond Google (e.g. DALL-E via OpenAI, Stability AI) with its own API key
- **EXTPROV-02**: User can configure an Anthropic Claude API key directly without going through OpenRouter
- **EXTPROV-03**: User can see cost estimates for OpenRouter runs with dollar amounts (requires OpenRouter `/models` pricing endpoint)

### Tooling

- **TOOL-01**: Provider configuration is validated at startup and shows a clear summary of active provider/model before generation begins
- **TOOL-02**: `--list-models` CLI flag shows available models for configured provider

## Out of Scope

| Feature | Reason |
|---------|--------|
| OpenRouter for image generation | OpenRouter image response format (`response.choices[0].message.images`) is incompatible with Gemini's `inline_data` format — unifying them is a rewrite risk; Gemini image quality is the product's differentiator |
| Abstract provider base classes / LiteLLM | Two providers do not justify ABC or registry patterns; flat `if/elif` is correct scope and maintainable |
| Web UI or API server | CLI/skill interface is the contract; GUI adds unrelated scope |
| Native DALL-E / Stable Diffusion image generation | Different SDK entirely; deferred to v2 |
| Video / animation output | Static infographics only |
